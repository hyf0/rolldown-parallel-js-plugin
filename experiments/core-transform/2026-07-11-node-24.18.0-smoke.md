# Core Transform Smoke and Failure Evidence

This is behavior evidence, not a performance benchmark. Single observed durations are retained only to identify the run and must not be used as speed claims.

## Environment

- Machine: Apple M3 Pro, 12 logical CPUs, macOS Darwin 25.5.0 arm64.
- Runtime: Node.js 24.18.0 installed and invoked explicitly through `mise exec node@24.18.0`.
- Package manager: pnpm 11.9.0 through the Node.js 24.18.0 Corepack shim.
- Rust: 1.97.0 (`2d8144b78`, 2026-07-07).
- Rolldown control: `21d7b32827045e377a82c3cb681dafa51c244883`.
- Research repair 1: `75ba695d1`, keep the worker event loop alive after successful bootstrap.
- Research repair 2: `8fe749827`, await every worker initialization result and terminate all peers after any failure; this commit also adds the three failure fixtures.
- Worker count: current default `min(os.availableParallelism(), 8)`, therefore 8 on this machine.

## Unchanged current main

Command from `examples/par-plugin`:

```sh
mise exec node@24.18.0 -- pnpm rolldown -c ./cases/noop-threejs10x/par-noop.rolldown.config.js
```

Observed result: exit 1 after 0.73 seconds of command wall time. The CLI reported two empty binding errors:

```text
ERROR  Build failed with 2 errors:

Error:
Error
Error:
Error
```

Repeating with `NODE_DEBUG=worker` showed all eight workers being created, entering their worker scripts, and then emitting their end events before the build error. This reproduces the worker-lifetime failure on current main and Node.js 24.18.0.

The ordinary control succeeded:

```sh
mise exec node@24.18.0 -- pnpm rolldown -c ./cases/noop-threejs10x/single-noop.rolldown.config.js
```

It reported 6.47 seconds, produced 10,167,549 bytes, and had SHA-256 `8227952f5da912ca7d095a0b08d03ac3f08a5e6a1b969f53b6c49de72bf68385`.

## Repair 1: worker lifetime

Commit `75ba695d1` retains the existing weak callback type and adds one long-lived timer after successful worker bootstrap. The main thread already terminates every worker explicitly when the build finishes.

The unchanged parallel no-op command then exited 0, reported 5.10 seconds on the first repaired run, and produced the same 10,167,549 bytes and SHA-256 as the ordinary control. A later warm-cache smoke run reported 3.81 seconds and the same hash. These unordered single runs prove behavior only, not a speedup.

The existing Babel/Rome controls also matched exactly:

| Variant | CLI duration | Bytes | SHA-256 |
| --- | ---: | ---: | --- |
| Ordinary Babel | 4.38 s | 2,682,699 | `cafb11e5747e1d190041cfb385cfd881d1c598cc54dcecb2bf3b903cf21ba888` |
| Parallel Babel | 4.38 s | 2,682,699 | `cafb11e5747e1d190041cfb385cfd881d1c598cc54dcecb2bf3b903cf21ba888` |

The parallel Babel result remained byte-identical after repair 2. A later warm-cache smoke run reported 3.08 seconds.

## Failure behavior before repair 2

The failure fixture accepts `PAR_FAILURE_MODE=init`, `sync`, or `reject`:

```sh
PAR_FAILURE_MODE=init mise exec node@24.18.0 -- pnpm rolldown -c ./cases/failure/failure.rolldown.config.js
```

Before repair 2, the init case first reported the expected `parallel-plugin-init-failure` and its worker-side source location, then aborted in another worker while the Node-API binding was still registering:

```text
ERROR  parallel-plugin-init-failure

    at parallel-failure-plugin/impl.js:5:11

Rolldown panicked. This is a bug in Rolldown, not your code.
thread '<unnamed>' panicked at napi-3.10.3/src/error.rs:698:1:
Throw error failed, status: [PendingException], raw message: "Create Custom GC Function in napi_register_module_v1 failed"
thread caused non-unwinding panic. aborting.
[ERR_PNPM_RECURSIVE_EXEC_FIRST_FAIL] Command was killed with SIGABRT (Aborted)
```

The process returned in 0.94 seconds but did not fail cleanly. The cause is the first rejecting `Promise.all` result allowing the main initialization path to unwind while peer workers are still importing and registering the binding.

## Repair 2: failed initialization cleanup

Commit `8fe749827` changes worker initialization to `Promise.allSettled`, awaits failed-worker termination, terminates any successful peers, and only then throws the first error.

The same init command now exits 1 in 0.42 seconds, retains `parallel-plugin-init-failure` and its worker source stack, and produces no panic or abort.

The synchronous and rejected transform fixtures also exit promptly:

| Mode | Command wall time | Exit | Retained error text |
| --- | ---: | ---: | --- |
| `sync` | 0.48 s | 1 | `parallel-plugin-sync-transform-failure` |
| `reject` | 0.50 s | 1 | `parallel-plugin-rejected-transform-failure` |

Both transform errors are flattened to `Error: Error: <message>` and lose the plugin name, module ID, and worker-side stack. This is a separate observability defect rather than a lifecycle failure.

## Validation notes

- `just build-glue` passed after both TypeScript repairs.
- The staged-file `vp check --fix` commit hook passed for both commits and the new fixtures.
- `just lint-node` reported every file correctly formatted, then failed on four pre-existing missing `@rolldown/test-dev-server` declarations in the fresh worktree. Those files and the missing package are unrelated to the ParallelPlugin changes.
- The primary Rolldown checkout and its pre-existing `test262` modification were not touched.

## Current conclusion

The retained transform architecture is callable on Node.js 24.18.0 after an explicit worker-lifetime owner is added. Weak thread-safe functions are not independently uncallable: they work while the worker event loop remains alive. Current main is nevertheless unusable unchanged because every worker exits after bootstrap. Successful transform output can match the ordinary plugin byte for byte, but initialization failure cleanup and transform error attribution are separate correctness concerns.
