# Cloudflare Docs MDX Results, 2026-07-12

## Benchmark boundary

Every performance conclusion in this document comes from processes launched on the local M3 Pro, not from a CI service. The direct runners record the default Cloudflare profile and `RUN_LINK_CHECK=false`; the original Astro reference additionally records `CI=null`. The ten-block kernel runner did not persist the `CI` variable itself, so this document does not claim artifact-level proof that it was unset. No timing from the link-check profile or any CI run is benchmark evidence, and the newer graph matrix runner refuses active CI markers.

The source is Cloudflare Docs at `2b08a67a41da1a521aecbcf465893abae1e9a6df`. The corpus contains 9,157 production content MDX modules totaling 32,619,831 source bytes: 6,719 docs, 1,449 partials, 988 changelog entries, and one compatibility-flags entry. Its manifest SHA-256 is `84077a08f660782274d5502be25f0ec9297cec9c52508e2c5e9e2a3e8bedc12b`.

The measurements used Node.js `v24.18.0` on an Apple M3 Pro with 12 logical CPUs and 36 GiB of RAM. The direct adapter uses the candidate's pinned Astro and MDX configuration, initializes `@mdx-js/rollup` and `@astrojs/mdx-postprocess`, and calls Rolldown directly. It does not run Vite or construct a Vite module graph. The Rolldown research runtime is `0aa600b5721b852cdc4095c7122a929a8cb4a798`, with binding SHA-256 `deec0b2cb7a12e507ff223e12535c3280ab5fe8371f2fcc92f9db206163f1c5d` and distribution SHA-256 `e30311e764bae7fba9afe27665db741d556a7c3728eb67cfbe7ce0fed3135ebc`.

## Repeated transform-kernel result

The transform kernel makes all 9,157 MDX modules Rolldown entries, runs the production MDX transform chain, and externalizes imports emitted by MDX. This isolates the real high-frequency JavaScript transform workload but is not a full application build.

The repeated local comparison used ten rotated blocks. Each block ran the ordinary main-thread plugin, a plugin-owned four-worker `node:worker_threads` pool, and Rolldown's four-worker ParallelPlugin in fresh processes. Instrumentation was disabled, samples had a 15-second cooldown, and all 30 runs completed successfully. Every run emitted 9,157 chunks and, after the narrow nondeterminism normalization described below, 166,149,564 bytes with SHA-256 `1840488b4a0d37a15851ea5dbc41e33f17cb1e9722b23c3048535d4d62274484`.

| Variant | Samples | Median wall | Mean wall | Wall range | Median CPU | Median peak RSS |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Ordinary main thread | 10 | 63.089 s | 63.883 s | 60.308–67.560 s | 78.956 s | 3.126 GB |
| Plugin-managed workers ×4 | 10 | 26.779 s | 27.940 s | 23.014–43.915 s | 130.994 s | 3.938 GB |
| Rolldown-managed workers ×4 | 10 | 28.226 s | 28.939 s | 24.954–36.703 s | 129.624 s | 4.133 GB |

The paired ordinary-to-plugin-managed median wall speedup was 2.386x, with a bootstrap 95% interval of 2.234–2.578. Its median CPU ratio was 1.638 and its median peak-RSS ratio was 1.214.

The paired ordinary-to-Rolldown-managed median wall speedup was 2.178x, with a bootstrap 95% interval of 2.129–2.369. Its median CPU ratio was 1.648 and its median peak-RSS ratio was 1.359.

The plugin-managed wall time divided by the Rolldown-managed wall time had a median of 0.927 and a bootstrap 95% interval of 0.874–1.034. Because this interval crosses parity, these runs establish the value of independent JavaScript worker isolates but do not establish an additional wall-time advantage from Rolldown-owned scheduling.

The host violated the intended clean-run policy. It already had about 16 GiB of swap in use and persistent background CPU activity. Every measured process observed pageouts, four of the ten Rolldown-managed samples observed new swapouts, and several samples began above the background-CPU threshold; neither ordinary nor plugin-managed samples observed new swapouts. The observed paired wall intervals remain above 2x, but this is not a clean-host formal confirmation. Precise resource ratios and the comparison between the two worker owners require a restarted, quiet local host.

The raw matrix is in [`data/2026-07-12-formal-kernel-10-blocks.raw.json`](data/2026-07-12-formal-kernel-10-blocks.raw.json), and the reconstructed statistics are in [`data/2026-07-12-formal-kernel-10-blocks.summary.json`](data/2026-07-12-formal-kernel-10-blocks.summary.json).

### Reproducibility limitation

The numerical summary can be reconstructed from the retained samples, but the historical run cannot be replayed exactly from the repository. The ten-block inputs did not record their runner source hash, parent CI-marker values, or `executionScope`. The two original Astro provenance records reference earlier runner hashes whose exact source text was not retained. Current matrix runners refuse active CI markers and record local-only scope, their own source hash, and the effective environment; the summary also marks this historical matrix benchmark-ineligible because its scope is unrecorded and every sample violated host policy. Clean-host formal confirmation requires a local rerun after restarting the host.

## Worker count and sustained execution

A local one-shot screening run measured 59.47 s for ordinary execution, 62.26 s for one worker, 38.63 s for two workers, 25.01 s for four workers, and 27.80 s for eight workers. One worker did not repay isolation and transport overhead. Four workers were best on this 12-logical-CPU host, while eight workers regressed and also used more CPU and memory. The screen does not isolate which of CPU competition, memory pressure, scheduling, JIT, or another interaction caused the regression. These values select a worker count; they are not a repeated confidence result.

The separate full instrumented run observed exactly 9,157 distinct transform IDs and 9,157 value results. It found no missing, duplicate, or unknown IDs and no transform errors or cancellations. At shutdown, queued, outstanding, and in-flight counts were all zero.

Four actual plugin factories were initialized. Pool readiness took 1.997 s, and each worker spent about 1.94–1.95 s constructing its production compiler instance. The workers processed 2,367, 2,398, 2,350, and 2,042 modules. The worker pool copied 32,619,831 input bytes into transforms and returned 167,169,554 bytes of code. Rolldown exposed almost the entire workload immediately: the permit queue peaked at 9,146 tasks, wrapper outstanding work peaked at 9,150, and at most four transforms were in service.

Initialization is measurable and makes a one-worker configuration slower, but about two seconds is not the limiting cost for this 9,157-module workload. The observed sustained costs are roughly 65% more total CPU and higher memory use. Compiler and cache duplication are direct architectural costs; garbage collection, memory bandwidth, and contention with Rolldown's Rust threads remain explanations to measure rather than causes proven by this experiment.

The ordinary transform handler reached 701 overlapping asynchronous calls. Their summed elapsed time is not CPU time because awaited work overlaps, so it must not be used to estimate the transform share of the application build. The instrumented evidence is in [`data/2026-07-12-full-instrumentation.raw.json`](data/2026-07-12-full-instrumentation.raw.json).

## Graph-preserving full scan

The graph adapter retains all project-local server-side imports that its strict resolver finds in code emitted from the MDX corpus, including aliases, relative and source-root paths, JavaScript, TypeScript, TSX, JSON, raw imports, assets, local CSS leaves, MDX, and Astro components. An unresolved project-local import fails the run. Bare packages, Astro runtime imports, and Node.js built-ins remain external. This resolver is experiment-owned: it does not execute the production MDX plugin's `resolveId`, including that hook's `this.resolve` and `./${source}` retry, so production resolver-context behavior is outside the graph claim.

The full scan reached 11,105 modules and 26,357 static edges: 6,907 project-local edges, 19,447 external edges, and three internal runtime edges. It included 9,157 MDX modules, 137 Astro module instances from 110 physical Astro files, 62 JavaScript modules, 1,733 asset leaves, 11 raw leaves, four data modules, and one CSS leaf. All three variants had zero unresolved local edges, the same graph and boundary hashes, 9,232 generated chunks, and the same 177,481,238-byte normalized output with SHA-256 `ccd4c151531c0877150f7375bc83e6d24ae130ba9e619ff317bbb00b4e6a29f4`.

This is a Cloudflare MDX/server graph, not the full Astro application. It excludes Astro's CSS virtual-module graph and 54 compiled CSS blocks, 27 client-script blocks, nine client specifier edges, 26 hydrated components, route rendering, and final asset emission. The boundary and exact counts are recorded in [`data/2026-07-12-graph-full-scan.raw.json`](data/2026-07-12-graph-full-scan.raw.json).

The one-shot local wall times were 65.462 s ordinary, 24.924 s plugin-managed with four workers, and 43.007 s Rolldown-managed with four workers. Their total CPU times were 84.112 s, 132.463 s, and 161.314 s, and their peak RSS values were 3.873 GB, 4.772 GB, and 3.721 GB. These are non-formal, single-sample results. They demonstrate that the preserved graph can complete with normalized output parity, but they do not support a performance comparison. In particular, the 43.007-second Rolldown-managed result conflicts with the repeated kernel behavior and must be repeated before it can be attributed to graph interaction or scheduling rather than host noise.

A local-only five-block graph matrix was attempted after adding an active-CI refusal and per-sample host counters. It was deliberately interrupted without writing a partial timing report when the fourth block's worker sample remained active after 103 seconds, host load reached 38.51, free memory fell to about 56 MiB by page count, and the already swap-heavy host continued thrashing. Completed in-memory samples were discarded. The [abort record](data/2026-07-12-graph-local-5-blocks.aborted.json) is resource evidence only and leaves repeated graph performance unresolved until the local host is restarted.

## Original local Astro build reference

One uninstrumented local default-profile `astro build` completed in 689.01 s wall time. Astro reported 8,650 pages built in 11m25s. The server Vite/Rollup bundling phase took 4m40s, and static route generation took 6m17s; content and build-info setup took 11.77 s, the client build took 12.21 s, and cached generation of 1,632 image outputs took 495 ms. `/usr/bin/time` reported 1,013.31 s user CPU, 78.27 s system CPU, 7.86 GB maximum RSS, and zero process swaps.

A separate instrumented local default-profile build observed exactly 9,157 MDX transform-handler calls over 9,157 distinct IDs, with 9,157 completions and zero failures. The sorted observed IDs exactly match all selected production-content MDX paths; both lists contain 1,417,724 bytes and have SHA-256 `e2c96d7aae510582c31e04b3fcb43afc41f44edf21b8aa6d08e052df9658ac54`. This turns the source-derived corpus size into an observed production-handler count. The handler window spanned about 71.8 s, maximum asynchronous overlap was 183, and summed handler elapsed was about 461.8 s. These values are coverage and concurrency evidence only: instrumentation and warm cache state changed the run, and awaited work overlaps, so its time is excluded from every benchmark comparison. The [ID-set validation](data/2026-07-12-astro-mdx-id-set-validation.json) and counter artifacts under [`data/2026-07-12-astro-local-mdx-counter-run`](data/2026-07-12-astro-local-mdx-counter-run/) retain the evidence.

This is a one-run end-to-end reference, not a comparison between Astro and Rolldown. The direct kernel's median reduction from 63.089 s to 28.226 s saves about 34.86 s. Transferring that saving exactly into the 689.01-second application build would illustrate only about a 5% reduction, not an integrated speedup or upper bound, because the adapter has not proved a one-to-one mapping from its kernel time into Astro's server bundling phase. Under the adapter boundary, the untouched 377-second static-route phase alone exceeds half of the complete build, so accelerating only the adapted transform stage cannot halve this workload. The reference artifacts are under [`data/2026-07-12-astro-local-reference`](data/2026-07-12-astro-local-reference/).

## Correctness and product gaps

### Returned metadata is lost

The production MDX transforms return `meta.astro` and `meta.vite`. The graph runner directly counted `meta.astro`: ordinary and plugin-managed execution retained it for all 9,157 MDX modules, while current Rolldown-managed ParallelPlugin execution retained it for zero. `meta.vite` travels through the same returned `meta` object but was not counted independently. Equal normalized generated code does not make this behavior equivalent. Metadata needs an explicit worker-to-coordinator result protocol.

### Plugin-global state becomes worker-local

Cloudflare's `starlight-links-validator` rehype plugin writes per-module headings and links into a `globalThis` Map. In a separate exploratory `RUN_LINK_CHECK=true` profile, ordinary execution left 9,157 entries in the main isolate, while both plugin-managed and Rolldown-managed worker variants left zero. The generated code still matched because the missing Map is consumed by a later validation step, not represented in each module's returned code.

That defect check was executed locally with `RUN_LINK_CHECK=true` solely to expose behavior used by the public CI configuration. Its timings are discarded and do not appear in any benchmark conclusion. A correct parallel design must return per-module validation state and merge it deterministically in the coordinator, or keep that stateful part of the plugin on the main thread.

### Production output contains random multipart boundaries

Cloudflare's Workers Playground plugin creates `FormData`, and Node.js Undici generates a random multipart boundary. Six source files with real `playground` fences emit seven dynamic Playground URLs, so raw output byte counts and hashes differ across otherwise equivalent runs. Correctness comparison decodes only those seven source-gated URLs, replaces the boundary, and re-encodes them. Raw outputs remain recorded. This is production nondeterminism, not worker computation divergence, but a general parallel-plugin system cannot assume that code equality is always a usable correctness oracle.

### Diagnostics, state, source maps, and failures remain open

The direct runners currently silence ordinary build diagnostics, while the original build reports MDX, unused-import, `eval`, route-conflict, missing-entry, and plugin-specific warnings. Parallel execution must preserve the plugin name, hook, module ID, location, stack, warning order where observable, and `this.error` and `this.warn` behavior. Worker crashes, task failures, cancellation, and cleanup must fail the build without losing the original diagnostic context or leaving work running.

Arbitrary closures, module singletons, caches, resolver context, and lifecycle state cannot be shared transparently across isolates. Worker-local caches are valid only if assignment and worker count affect performance but never output or diagnostics. Stateful plugins need an explicit deterministic reduction step. Production source maps in this corpus were null, so this experiment does not establish source-map-chain correctness for multiple transforms.

## Conclusion

Parallel JavaScript plugins have strong repeated local value evidence when a build has thousands of mandatory, expensive, module-local JavaScript transforms ready at the same time. Four workers reduced the Cloudflare MDX kernel from a 63.089-second median to 28.226 seconds under Rolldown management, and a plugin-owned pool achieved a similar 26.779 seconds. The measured value comes from independent Node.js worker isolates. The host-policy failures still require clean-host confirmation, and this study has not shown that current Rolldown-owned scheduling is faster than a well-built plugin-owned pool.

Cloudflare Docs is a positive transform-stage example but a negative whole-build example for a 15→7–8 minute or 30→15 minute target. Its static route work is larger than the transform kernel, so a 2.178x stage speedup cannot halve the full build. Module count alone is insufficient; the required JavaScript transform chain must occupy most of the direct-Rolldown critical path.

Current ParallelPlugin is not production-compatible for this corpus because it loses returned metadata and isolate-local plugin state, and because diagnostics, errors, lifecycle state, cache determinism, source-map chains, and failure behavior are not yet equivalent to ordinary plugins.

## Next iteration

1. Repeat the graph-preserving ordinary, plugin-managed, and Rolldown-managed comparison in rotated local blocks on a restarted, quiet host, retaining host CPU, pageout, swapout, RSS, and per-worker throughput measurements.
2. Define a coordinator protocol for returned metadata, deterministic per-module state reduction, warnings, errors, and worker crashes, then turn the measured metadata and link-validator failures into conformance tests.
3. Compare a Rolldown-managed shared worker group, an explicitly exclusive worker group, and plugin-owned pools under the same combined Rust-thread and JavaScript-worker CPU budget.
4. Test whether several high-frequency JavaScript transforms can run in hook order inside one worker for each module, avoiding repeated cross-thread transport while preserving source-map chains, metadata, diagnostics, error attribution, and deterministic caches.
5. Measure lazy clean-build initialization, JIT warmup, sustained per-worker throughput, garbage collection, RSS, and memory bandwidth separately from task scheduling. Cross-build worker reuse remains outside this no-watch scope.
6. Select the next direct-Rolldown application by measured transform share rather than module count, then apply the same protocol to a Vue project after the core ParallelPlugin flow is correct.
