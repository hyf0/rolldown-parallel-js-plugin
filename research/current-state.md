# Current-State Evidence

Snapshot date: 2026-07-11. This file records source-backed facts and open checks before new implementation work.

## Current conclusion

The parallel JavaScript plugin implementation has not been deleted, but current main is known not to work. [PR #2135](https://github.com/rolldown/rolldown/pull/2135) deliberately made every JavaScript callback use a weak Node-API thread-safe function to fix process exit and explicitly stated that this breaks `ParallelPlugin`; Node workers cannot use that weak lifetime mode. The useful question is no longer whether the retained prototype once invoked a JavaScript plugin from several workers; it is whether the runtime can be restored with clear lifecycle semantics, whether a real stateful plugin can be adapted without losing behavior, and whether end-to-end gains in `resolveId`, `load`, or `transform` survive overhead and contention with the concurrent Rust core.

## Rolldown at `025ebaf18db8022f18c845f37cfca56d0cb332ea`

- [`defineParallelPlugin`](https://github.com/rolldown/rolldown/blob/025ebaf18db8022f18c845f37cfca56d0cb332ea/packages/rolldown/src/plugin/parallel-plugin.ts) is still exported through `rolldown/experimental`. The worker-side factory is exposed from the separate `rolldown/parallelPlugin` entry.
- [`initializeParallelPlugins`](https://github.com/rolldown/rolldown/blob/025ebaf18db8022f18c845f37cfca56d0cb332ea/packages/rolldown/src/utils/initialize-parallel-plugins.ts) creates one Node.js worker per reported CPU up to eight whenever at least one parallel plugin exists. It sends the plugin module URL and structured-cloned options to every worker and initializes every parallel plugin in every worker.
- [`parallel-plugin-worker.ts`](https://github.com/rolldown/rolldown/blob/025ebaf18db8022f18c845f37cfca56d0cb332ea/packages/rolldown/src/parallel-plugin-worker.ts) imports each implementation module, constructs the plugin with a `threadNumber`, converts it to binding callbacks inside that worker's Node.js environment, and registers those callbacks in a process-wide Rust registry.
- [`ParallelJsPlugin`](https://github.com/rolldown/rolldown/blob/025ebaf18db8022f18c845f37cfca56d0cb332ea/crates/rolldown_binding/src/options/plugin/parallel_js_plugin.rs) stores one `JsPlugin` per worker. Per-module throughput hooks acquire one worker permit and call the corresponding instance; lifecycle or notification hooks use a whole-pool permit and call every instance.
- [`WorkerManager`](https://github.com/rolldown/rolldown/blob/025ebaf18db8022f18c845f37cfca56d0cb332ea/crates/rolldown_binding/src/worker_manager.rs) is shared by all parallel plugins in one bundler, so their calls contend for the same fixed pool. `run_all` temporarily reserves the entire pool.
- The Rust module loader spawns concurrent Tokio tasks for discovered modules, so different modules can reach plugin hooks concurrently. Hook order remains sequential for a single module.
- Ordinary JavaScript plugin functions are also represented as Node-API thread-safe functions, but those callbacks belong to the main Node.js environment. Parallel plugin callbacks belong to worker environments, which is the mechanism that moves CPU-bound JavaScript away from the main event loop.
- The current callback alias sets the worker-incompatible weak mode. [Rolldown issue #1003](https://github.com/rolldown/rolldown/issues/1003) proposed separate main and worker callback types but was closed in April 2025 because parallel JavaScript plugins were not an active investment; the blocking [napi-rs issue #2071](https://github.com/napi-rs/napi-rs/issues/2071) remains open.
- Workers are initialized for each `generate` or `write` call and stopped when the next build begins or the build object closes. Watch mode retains its workers. Reuse and lifecycle behavior in the newer DevEngine path need a dedicated check.
- The Rust adapter currently forwards nine hooks: `buildStart`, `resolveId`, `load`, `transform`, `moduleParsed`, `buildEnd`, `renderChunk`, `generateBundle`, and `writeBundle`. The underlying JavaScript binding supports additional hooks that silently fall through in `ParallelJsPlugin`; [PR #9318](https://github.com/rolldown/rolldown/pull/9318) added only `moduleParsed` in May 2026 and explicitly noted the remaining gap.
- Parallel markers are excluded from main-thread `options`, `outputOptions`, and `onLog` processing. Output-option plugins are marked unsupported, and Vite-specific hooks are outside the current adapter. The worker constructs `PluginContextData` with empty input and output options, empty plugin lists, a no-op logger, and no shared instance, so contract compatibility cannot be assumed from the ordinary `Plugin` type.
- Worker-local `PluginContextData` means custom module metadata and JavaScript-side caches are not automatically shared across instances. A later hook may run on another worker unless the runtime adds affinity or the plugin externalizes that state.
- Automated coverage is weak: current main has examples and a Babel bench but no committed integration test found for the parallel plugin path. [PR #9318](https://github.com/rolldown/rolldown/pull/9318) merged without the proposed integration test.
- [PR #4723](https://github.com/rolldown/rolldown/pull/4723) restored the Babel benchmark in May 2025 while explicitly noting that parallel plugins did not work. The approving review on [PR #9318](https://github.com/rolldown/rolldown/pull/9318) said development was paused while the project focused elsewhere.

## History and prior evidence

- [PR #825](https://github.com/rolldown/rolldown/pull/825) introduced the prototype in April 2024. Its Babel result reported a parallel build about 1.83 times faster than the single-thread form, while lightweight Sucrase work regressed because worker creation and repeated V8 parsing or compilation dominated.
- The precursor [`parallel-js-plugin-experiment`](https://github.com/sapphi-red/parallel-js-plugin-experiment/tree/365de4f16e9acd83fcd5bfd532ccf3e906459e9e) measured roughly 26–57 ms to start 1–16 workers on its machine and showed that direct Rust-to-worker Node-API callbacks avoided the very large message-port overhead of an indirect main-thread relay. Its plugin survey already identified module metadata, per-file affinity, shared caches, lifecycle broadcast, non-serializable options, direct plugin communication, and Vite-only hooks as the main design problems.
- Rolldown's unmerged branch [`feat/native-bridge-plugin-vue-bench`](https://github.com/rolldown/rolldown/tree/8aed488c69d8e2a3b6734d2fbc5e8635a42c8307) contains newer React Compiler and Vue SFC bridge experiments. It has no associated pull request and is based on `bea835c94312169a560c24b2ce457a380f32640c`, so it is evidence to audit rather than a proposal to continue blindly.
- The branch's [Vue result](https://github.com/rolldown/rolldown/blob/8aed488c69d8e2a3b6734d2fbc5e8635a42c8307/scripts/bench/seven-way-vue/results.md) reports 50.1 ms for an eight-worker Vize bridge versus 33.2 ms for the sync Vize bridge on 255 Elk SFCs, while a heavier 3,176-file React Compiler corpus shows the worker form far ahead of the sync bridge. This supports a workload-dependent crossover, not a general verdict.
- That Vue benchmark is not a real `@vitejs/plugin-vue` result: it externalizes non-SFC dependencies, compiles SFCs from a virtual entry, omits style compilation, uses Vize for the worker variant, and replaces compiler failures with `export default {}`. Its JavaScript `@vue/compiler-sfc` variant is useful as an isolated compiler baseline only.
- The same branch records a Node.js 24 `Status::Closing` failure after worker bootstrap and adds an interval to keep workers alive. Current main still has the pre-patch worker exit shape. This is an open reproduction check because the branch patch was not merged and later Node-API changes may affect the outcome.
- The branch's React result document contains stale internal contradictions about whether the async variants still deadlock at full corpus. Raw runs and a clean current reproduction are required before reusing any such claim.

## Current Vue plugin at `28d6889104e76c8f420910bee6d17a081b130f1d`

- [`@vitejs/plugin-vue`](https://github.com/vitejs/vite-plugin-vue/blob/28d6889104e76c8f420910bee6d17a081b130f1d/packages/plugin-vue/src/index.ts) is one plugin object containing Vite configuration, HMR, server setup, `buildStart`, `resolveId`, `load`, and `transform` behavior.
- Its SFC pipeline uses process-local descriptor, HMR, previous-descriptor, client script, server script, and type-dependency caches. Main SFC transforms populate descriptors that later virtual script, template, style, and custom-block requests consume.
- [`descriptorCache.ts`](https://github.com/vitejs/vite-plugin-vue/blob/28d6889104e76c8f420910bee6d17a081b130f1d/packages/plugin-vue/src/utils/descriptorCache.ts) and [`script.ts`](https://github.com/vitejs/vite-plugin-vue/blob/28d6889104e76c8f420910bee6d17a081b130f1d/packages/plugin-vue/src/script.ts) therefore make naive worker replication unsafe: related hooks can land on different workers, plugin-pretransformed source may differ from a fresh file read, and HMR needs prior state.
- The initial build experiment can avoid HMR but still needs resolved configuration, descriptor ownership, virtual submodule behavior, source maps, watch dependencies, and plugin-context calls to remain correct.

## Current Svelte plugin at `c9feef65f78bb42a1dc85c58522ca3c9bdfde01c`

- [`@sveltejs/vite-plugin-svelte`](https://github.com/sveltejs/vite-plugin-svelte/blob/c9feef65f78bb42a1dc85c58522ca3c9bdfde01c/packages/vite-plugin-svelte/src/index.js) now returns a list of task-specific plugin objects sharing a small API object. Configuration, preprocessing, compilation, compiled-CSS loading, HMR, and optimizer work are already separated.
- The [`compile` plugin](https://github.com/sveltejs/vite-plugin-svelte/blob/c9feef65f78bb42a1dc85c58522ca3c9bdfde01c/packages/vite-plugin-svelte/src/plugins/compile.js) is close to a module-local worker kernel: it parses the id, compiles one source, returns JavaScript and CSS through module metadata, and reports warnings.
- The split does not make it immediately parallel-safe. Resolved options can contain non-serializable preprocessors, `dynamicCompileOptions`, and warning handlers; compile statistics are shared mutable state; compiled CSS is later read through module metadata; preprocessing tracks dependency maps; HMR maintains prior transformed outputs.
- Svelte is therefore the provisional first adaptation because the heavy compile boundary is already visible. It is also a direct test of whether Rolldown can transport declared metadata and diagnostics from worker-local compilation back to coordinating hooks.

## Open checks before implementation

- Reproduce the known weak-callback failure and the separate Node.js 24 worker keepalive failure on supported Node.js versions before deciding the smallest viable restoration path.
- Enumerate every plugin context method under a worker and prove which state is Rust-owned, worker-local, main-thread-local, or unsupported.
- Profile and model `resolveId`, `load`, and `transform` separately; the current evidence is heavily transform-biased and cannot support hook-general conclusions.
- Confirm hook filters are compiled and applied consistently for parallel instances and quantify the calls they eliminate.
- Decide whether the first real build targets Rolldown directly or Vite's Rolldown integration; the latter is necessary for full official plugin behavior but introduces Vite orchestration as another measured layer.
- Profile candidate Vue and Svelte projects before selecting them so the study does not choose a project with no plugin headroom or a synthetic project designed to favor parallelism.
