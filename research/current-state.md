# Current-State Evidence

Snapshot date: 2026-07-11. Rolldown source snapshot: `21d7b32827045e377a82c3cb681dafa51c244883`. This file records source-backed facts and the first Node.js 24.18.0 runtime results. The unchanged parallel-plugin paths are byte-identical to the earlier `025ebaf18db8022f18c845f37cfca56d0cb332ea` snapshot used during repository initialization.

## Current conclusion

The parallel JavaScript plugin implementation has not been deleted, but unchanged current main is not usable on Node.js 24.18.0: all eight workers exit after bootstrap and the first parallel build fails. Research commit `75ba695d1` adds an explicit keepalive, and `8fe749827` prevents a plugin initialization failure from leaving peer workers to abort with a Node-API pending exception. The formal release-binding evidence separates main-thread isolation from throughput: worker-1 makes the synthetic one-millisecond transform build about 9.9% slower but reduces maximum main-loop delay from about 1.025 seconds to 2.44 milliseconds; a wide heavy controlled graph reaches 4.19x at eight workers and 4.75x at twelve, while cheap calls, 32 modules, and a serial 512-module chain regress. Startup adds roughly 47–98 milliseconds in the near-empty case and minimal-plugin RSS grows about 12 MiB per worker. The observed crossover, saturation point, and gain depend on call cost, ready concurrency, worker count, payload, CPU, and per-isolate import or JIT work. [Controlled release report](../experiments/core-transform/2026-07-11-controlled-release.md), [runtime repairs](../experiments/core-transform/2026-07-11-node-24.18.0-smoke.md)

## Rolldown at `21d7b32827045e377a82c3cb681dafa51c244883`

- [`defineParallelPlugin`](https://github.com/rolldown/rolldown/blob/21d7b32827045e377a82c3cb681dafa51c244883/packages/rolldown/src/plugin/parallel-plugin.ts) is still exported through `rolldown/experimental`. The worker-side factory is exposed from the separate `rolldown/parallelPlugin` entry.
- [`initializeParallelPlugins`](https://github.com/rolldown/rolldown/blob/21d7b32827045e377a82c3cb681dafa51c244883/packages/rolldown/src/utils/initialize-parallel-plugins.ts) creates one Node.js worker per reported CPU up to eight whenever at least one parallel plugin exists. It sends the plugin module URL and structured-cloned options to every worker and initializes every parallel plugin in every worker.
- [`parallel-plugin-worker.ts`](https://github.com/rolldown/rolldown/blob/21d7b32827045e377a82c3cb681dafa51c244883/packages/rolldown/src/parallel-plugin-worker.ts) imports each implementation module, constructs the plugin with a `threadNumber`, converts it to binding callbacks inside that worker's Node.js environment, and registers those callbacks in a process-wide Rust registry.
- [`ParallelJsPlugin`](https://github.com/rolldown/rolldown/blob/21d7b32827045e377a82c3cb681dafa51c244883/crates/rolldown_binding/src/options/plugin/parallel_js_plugin.rs) stores one `JsPlugin` per worker. Per-module throughput hooks acquire one worker permit and call the corresponding instance; lifecycle or notification hooks use a whole-pool permit and call every instance.
- [`WorkerManager`](https://github.com/rolldown/rolldown/blob/21d7b32827045e377a82c3cb681dafa51c244883/crates/rolldown_binding/src/worker_manager.rs) is shared by all parallel plugins in one bundler, so their calls contend for the same fixed pool. `run_all` receives every permit one at a time and retains the whole pool until all broadcasts finish.
- The Rust module loader spawns concurrent Tokio tasks for discovered modules, so different modules can reach plugin hooks concurrently. It also resolves the distinct import records discovered in one module with `join_all`. Hook order remains sequential for a single plugin chain, and the first non-null `resolveId` or `load` result stops that chain.
- Ordinary JavaScript plugin functions are also represented as Node-API thread-safe functions, but those callbacks belong to the main Node.js environment. Parallel plugin callbacks belong to worker environments, which is the mechanism that moves CPU-bound JavaScript away from the main event loop.
- Per-hook calls do not travel through a main-thread `postMessage`. Rust calls the thread-safe function created in the selected worker environment directly. `workerData` structured-clones plugin options at initialization; hook values still require Node-API conversion into and out of the worker isolate.
- The current callback alias still sets `Weak = true`, the same configuration that PR #2135 reported as breaking worker use. [Rolldown issue #1003](https://github.com/rolldown/rolldown/issues/1003) proposed separate main and worker callback types but was closed in April 2025 because parallel JavaScript plugins were not an active investment; the supporting [napi-rs issue #2071](https://github.com/napi-rs/napi-rs/issues/2071) remains open.
- Workers are initialized for each `generate` or `write` call and stopped when the next build begins or the build object closes. Watch mode retains its workers, but watch and cross-build reuse are outside the confirmed research scope.
- The Rust adapter currently forwards nine hooks: `buildStart`, `resolveId`, `load`, `transform`, `moduleParsed`, `buildEnd`, `renderChunk`, `generateBundle`, and `writeBundle`. The underlying JavaScript binding supports additional hooks that silently fall through in `ParallelJsPlugin`; [PR #9318](https://github.com/rolldown/rolldown/pull/9318) added only `moduleParsed` in May 2026 and explicitly noted the remaining gap.
- The wrapper returns `HookUsage::all()` but does not forward any hook-order metadata. Unsupported hooks therefore appear available, and `order: 'pre' | 'post'` on implemented worker hooks does not affect the plugin driver's ordering.
- `resolveId`, `load`, `transform`, and `renderChunk` filters are still compiled into each selected `JsPlugin` and can avoid the JavaScript callback. The wrapper acquires a worker permit before that filter check, so filters remove bridge and plugin work but do not remove pool queueing and selection overhead.
- Parallel markers are excluded from main-thread `options`, `outputOptions`, and `onLog` processing, are not included in the `RolldownOutputPlugin` type, and have no Vite-specific hook path. The worker constructs `PluginContextData` with empty input and output options, empty plugin lists, a no-op logger, and no shared instance, so contract compatibility cannot be assumed from the ordinary `Plugin` type.
- Every plugin in every worker receives its own `PluginContextData`. Rust-backed graph structure, file emission, module loading, resolution, and watch-file registration are shared, but JavaScript module metadata, custom resolver option receipts, load promises, normalized option views, and logs are not. A later hook may run on another worker unless the runtime adds affinity or the plugin externalizes that state.
- `generateBundle` and `writeBundle` each select one available instance, so neither can reconstruct state accumulated across all instances without an explicit reduction. `moduleParsed` instead broadcasts every module to every instance, multiplying callbacks by the worker count and blocking the entire pool during each broadcast.
- Source audit identified several concrete compatibility failures and two unreproduced permit-cycle paths. They are tracked with evidence and verification status in [the defect inventory](./defect-inventory.md).
- Automated coverage is weak: current main has examples and a Babel bench but no committed integration test found for the parallel plugin path. [PR #9318](https://github.com/rolldown/rolldown/pull/9318) merged without the proposed integration test.
- [PR #4723](https://github.com/rolldown/rolldown/pull/4723) restored the Babel benchmark in May 2025 while explicitly noting that parallel plugins did not work. The approving review on [PR #9318](https://github.com/rolldown/rolldown/pull/9318) said development was paused while the project focused elsewhere.

## History and prior evidence

- [PR #825](https://github.com/rolldown/rolldown/pull/825) introduced the prototype in April 2024. A later corrected [single-build Babel script](https://github.com/rolldown/rolldown/pull/825#issuecomment-2054028148) reported a parallel build about 1.83 times faster than the single-thread form, while lightweight Sucrase work regressed because worker creation and repeated V8 parsing or compilation dominated.
- The precursor [`parallel-js-plugin-experiment`](https://github.com/sapphi-red/parallel-js-plugin-experiment/tree/365de4f16e9acd83fcd5bfd532ccf3e906459e9e) measured roughly 26–57 ms to start 1–16 workers on its machine and showed that direct Rust-to-worker Node-API callbacks avoided the very large message-port overhead of an indirect main-thread relay. Its plugin survey already identified module metadata, per-file affinity, shared caches, lifecycle broadcast, non-serializable options, direct plugin communication, and Vite-only hooks as the main design problems.
- Rolldown's unmerged branch [`feat/native-bridge-plugin-vue-bench`](https://github.com/rolldown/rolldown/tree/8aed488c69d8e2a3b6734d2fbc5e8635a42c8307) contains newer React Compiler and Vue SFC bridge experiments. It has no associated pull request and is based on `bea835c94312169a560c24b2ce457a380f32640c`, so it is evidence to audit rather than a proposal to continue blindly.
- The branch's [Vue result](https://github.com/rolldown/rolldown/blob/8aed488c69d8e2a3b6734d2fbc5e8635a42c8307/scripts/bench/seven-way-vue/results.md) reports 50.1 ms for an eight-worker Vize bridge versus 33.2 ms for the sync Vize bridge on 255 Elk SFCs, while a heavier 3,176-file React Compiler corpus shows the worker form far ahead of the sync bridge. This supports a workload-dependent crossover, not a general verdict.
- That Vue benchmark is not a real `@vitejs/plugin-vue` result: it externalizes non-SFC dependencies, compiles SFCs from a virtual entry, omits style compilation, uses Vize for the worker variant, and replaces compiler failures with `export default {}`. Its JavaScript `@vue/compiler-sfc` variant is useful as an isolated compiler baseline only.
- The same branch records a Node.js 24 `Status::Closing` failure after worker bootstrap and adds an interval to keep workers alive. Current main still has that pre-patch worker exit shape, and the failure is now reproduced on Node.js 24.18.0. Research commit `75ba695d1` confirms that the same timer-shaped repair restores the path.
- The branch's React result document contains stale internal contradictions about whether the async variants still deadlock at full corpus. Raw runs and a clean current reproduction are required before reusing any such claim.

## Archived official Vue plugin source at `28d6889104e76c8f420910bee6d17a081b130f1d`

- [`@vitejs/plugin-vue`](https://github.com/vitejs/vite-plugin-vue/blob/28d6889104e76c8f420910bee6d17a081b130f1d/packages/plugin-vue/src/index.ts) is one plugin object containing Vite configuration, HMR, server setup, `buildStart`, `resolveId`, `load`, and `transform` behavior.
- Its SFC pipeline uses process-local descriptor, HMR, previous-descriptor, client script, server script, and type-dependency caches. Main SFC transforms populate descriptors that later virtual script, template, style, and custom-block requests consume.
- [`descriptorCache.ts`](https://github.com/vitejs/vite-plugin-vue/blob/28d6889104e76c8f420910bee6d17a081b130f1d/packages/plugin-vue/src/utils/descriptorCache.ts) and [`script.ts`](https://github.com/vitejs/vite-plugin-vue/blob/28d6889104e76c8f420910bee6d17a081b130f1d/packages/plugin-vue/src/script.ts) therefore make naive worker replication unsafe: related hooks can land on different workers, plugin-pretransformed source may differ from a fresh file read, and HMR needs prior state.
- Its own `resolveId` is a query identity mapping, and ordinary `load` work is primarily helper code, descriptor lookup, or a script-cache hit. The main SFC `transform` is the likely CPU candidate; `resolveId` and `load` remain overhead and correctness controls.
- These findings explain why `@vitejs/plugin-vue` cannot be treated as a direct Rolldown plugin. The current second case instead uses a transform-only adapter around the real `unplugin-vue/rolldown` entry, whose internal compiler is `@vue/compiler-sfc`, and states which SFC behavior it supports.

## Archived official Svelte plugin source at `c9feef65f78bb42a1dc85c58522ca3c9bdfde01c`

- [`@sveltejs/vite-plugin-svelte`](https://github.com/sveltejs/vite-plugin-svelte/blob/c9feef65f78bb42a1dc85c58522ca3c9bdfde01c/packages/vite-plugin-svelte/src/index.js) now returns a list of task-specific plugin objects sharing a small API object. Configuration, preprocessing, compilation, compiled-CSS loading, HMR, and optimizer work are already separated.
- The [`compile` plugin](https://github.com/sveltejs/vite-plugin-svelte/blob/c9feef65f78bb42a1dc85c58522ca3c9bdfde01c/packages/vite-plugin-svelte/src/plugins/compile.js) exposes a promising module-local boundary only after a coordinator supplies the filter, options, ID parser, compile closure, Vite environment identity, and combined source map initialized or provided by the ordinary plugin lifecycle. It cannot be moved unchanged into the current ParallelPlugin.
- The split does not make it immediately parallel-safe. Resolved options can contain non-serializable preprocessors, `dynamicCompileOptions`, and warning handlers; compile statistics are shared mutable state; compiled CSS is later read through module metadata; preprocessing tracks dependency maps; HMR maintains prior transformed outputs.
- Its compiled-CSS `resolveId` is an exact-filtered constant return, and its `load` path depends on graph metadata and a rebuild cache. Those hooks should initially remain coordinated while the synchronous compiler kernel is the primary worker candidate.
- The isolated direct-Rolldown Svelte case uses one shared, stateless `svelte.compile` kernel over 1,340 pinned shadcn-svelte components. Four workers reduce paired median fixture wall time from 2054 ms to 1509 ms, a 1.36x speedup, while increasing user CPU from 3008 ms to 5689 ms and peak RSS from 748 MiB to 983 MiB. The 24-component fixture loses at every tested worker count, and eight or twelve workers are worse than four. [Formal Svelte result](../experiments/svelte-transform/2026-07-11-svelte-results.md)
- The isolated Svelte output code and maps match exactly, but the current worker context drops two structured compiler warnings. Worker errors lose plugin and hook attribution and retain only a relative filename and position instead of the complete module path. The synthetic fan-out makes every SFC a root and externalizes every SFC dependency, so this is a prepared compiler-kernel upper bound rather than representative-project evidence. A graph-preserving shadcn registry UI subgraph is active as the completion case.

The complete source-derived hook maps, serialization constraints, worker-boundary hypotheses, correctness fixtures, and candidate project pool are in [Vue and Svelte plugin case notes](./plugin-case-notes.md).

## `resolveId` and `load` evidence

- In the controlled release confirmation, 512 independent synchronous CPU-heavy calls reach paired median speedups of 1.50x, 2.32x, and 2.91x for `resolveId` at two, four, and eight workers, and 1.45x, 2.18x, and 2.74x for `load`. One worker regresses to about 0.85x for both but reduces event-loop p99 from roughly the full half-second ordinary build to 1.2–1.3 ms. [Controlled hook result](../experiments/resolve-load/2026-07-11/README.md)
- Cheap hooks, cached short synchronous file probes, 64 KiB returned load results without enough computation, and equally expensive serial chains all regress. An ordinary async load overlaps 512 five-millisecond timers in about 22 ms, while worker-1 takes about 3.0 seconds and worker-8 about 440 ms because each pending Promise retains a pool permit.
- The hook probes observe three compatibility defects: a native filter miss acquires a permit before rejection; mutable closure state is partitioned across worker instances and changes the output; and same-plugin `this.resolve(..., { skipSelf: false })` deadlocks with one worker while ordinary and two-worker forms complete with the same output. Resolve and load failures propagate without a signal or abort, but worker stderr loses the ordinary plugin label and original handler frame.

- No current `resolveId` plugin has yet qualified as a clean positive JavaScript-worker case. Vite's production resolver and tsconfig-path support now run through Rolldown's native resolver, while current `vite-tsconfig-paths` delegates per-import work to `oxc-resolver`. NativeScript platform-file probing and a direct Rolldown Yarn PnP plugin remain provisional synchronous candidates, but the former still needs a Vite 8 project match and the latter still needs a substantial independent consumer plus a native baseline.
- `@rollup/plugin-node-resolve` remains the strongest real resolver to profile and the most useful semantics case, but it mixes async filesystem work with JavaScript resolution, recursively calls `this.resolve` with custom receipts, exposes instance-local package information, and owns several caches. The current whole-plugin replication model cannot run it faithfully.
- The source survey found three credible `load` hypotheses in current Vite 8 projects: high-volume `vite-plugin-svgr` on ZenML Dashboard, lower-state `vite-svg-loader` on `vue-pure-admin`, and cache-heavy `unplugin-icons` on the same Vue project. SVGR's Vite 8 path also invokes synchronous native Oxc and therefore needs stage-level attribution.
- All three loader sources perform their match check inside a JavaScript function hook. Unplugin's `loadInclude` is also wrapped inside JavaScript rather than converted into a native Rolldown filter. A coordinator-side or declarative filter before worker dispatch is a common authoring requirement, and its standalone gain must be separated from worker value.

Pinned sources, static project inventories, and rejection reasons are retained in the [`resolveId` and `load` candidate survey](./resolve-load-candidates.md). The survey connects the controlled mechanisms to possible real plugins but is not itself performance evidence.
