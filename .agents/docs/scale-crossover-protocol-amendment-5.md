# Scale-Crossover Protocol Amendment 5: MDX Attribution and Deterministic Follow-up

Status: frozen on 2026-07-12 before the attribution lane, base screen, confirmation, or refinement is executed. This amendment changes only the MDX attribution runtime and the mechanical orchestration of already frozen MDX scale points. The lifecycle-fixed wall baseline, corpus, prefix order, host gate, correctness gate, statistical thresholds, resource thresholds, product semantics, and local-only scope remain unchanged.

## Attribution runtime and classification

The attribution-only runtime is Rolldown commit `8e35a2249b60b65120a44d1d896eeeed19dc703b`, binding SHA-256 `6b7dfa175754ac57650768a68d7a567c5c0635a1bb47d47c5287914594c9795e`, and package-dist SHA-256 `68f57be9a8883a4ca6f28b57a9bac6e16907d8c1d079686ab9921b407b132735`. A matrix may point to any absolute clean checkout containing that exact triple; the child rechecks clean Git HEAD, binding bytes, and the full dist tree before importing Rolldown. The checkout path is not evidence identity.

This runtime cannot provide wall, correctness, crossover, or optimum evidence. Every result is classified `attribution`, retains measurement fields, and sets both `timingEligible` and `conclusionEligible` to false. It may run only after the lifecycle-fixed correctness gate passes and each child passes the same frozen pre-run and post-run host gates. Host admission controls attribution quality but does not promote instrumented durations into benchmark evidence.

The formal matrix contains one fresh-process run each of ordinary, worker four, and worker eight over the exact 9,157-source prefix. It enables JavaScript and Rust instrumentation for every variant, including the process-level native module-initialization record in ordinary execution. Exactly one module-initialization record must appear per process. Worker variants must additionally emit exactly one transform record plus one initialization and one termination record; ordinary execution must emit none of those worker-only records.

Admission requires exact JavaScript handler identity and one hit per source; raw Rust call and arrival/acquire/complete event cardinality; a bijection between Rust permit indices, JavaScript per-module worker indices, and Node worker thread numbers; complete ready, busy, service, width, and throughput inputs; main-thread, each-worker, whole-process, and residual CPU; sampled and external peak RSS; main and worker heap, event-loop utilization, and garbage collection; clean queue, completion, lifecycle, and termination state; a post-child pageout/swapout gate; and normalized output equality with the pinned lifecycle-fixed correctness oracle. Missing, empty, duplicated, failed, or internally inconsistent captures reject the run.

## Deterministic screen follow-up

The committed generator consumes the exact passed base-screen artifact rather than a hand-copied table. At each confirmation scale it selects the fastest screened worker count, includes its lower and higher eligible adjacent counts, always includes worker four and worker eight as fixed policy candidates, removes duplicates, and emits ten no-warmup rotated blocks. A boundary worker count includes its one available neighbor. The generated matrix records the source screen SHA-256 and the ordered SHA-256 chain of every consumed follow-up artifact.

The first confirmation covers the lower endpoint of the selected interval, its upper endpoint, the next larger base point when one exists, and the full corpus, with duplicates removed. If the one-shot screen begins with a worker win, the first base interval is retained for possible left-censoring; if it has no worker win, the final base interval is retained for possible right-censoring. A screen remains work selection and never establishes a crossover.

Refinement is iterative. Before a new frozen refinement prefix is confirmed, that prefix receives one ordinary plus worker-one-through-eight screen. Its fastest worker, adjacent counts, worker four, and worker eight then receive ten rotated confirmation blocks. The next prefix is selected mechanically from the remaining points in the frozen first interval; artifacts that skip, reorder, repeat, or leave that interval are rejected. A positive-to-negative reversal in either the base screen or repeated refinement evidence is a hard rejection rather than permission to select the first favorable interval. The generator uses 100,000 deterministic paired-median bootstrap resamples with seed `0x20260712`, applies the previously frozen resource gates, and stops only with an exact mechanical/resource boundary or an explicit left-, right-, edge-, or interval-censored result. Exact means the candidate is repeatedly positive, its immediate smaller activated frozen point excludes an earlier crossover, and its immediate larger activated point confirms the direction.

Every repeated point emits a compact `policyEvidence` object keyed by exact variant. It records wall median, total-process CPU median, peak-RSS median, frozen resource eligibility, the paired worker-to-ordinary wall-ratio bootstrap 95% upper bound, and the selected resource-eligible oracle worker count. Candidate selection separately records each candidate wall-median bootstrap interval and uses those wall intervals, not paired speedup intervals, for the frozen smaller-count tie rule. The decision additionally exposes the compact blocks under `policyEvidenceByScale`, so later fixed-policy and quota evaluators can bind stable JSON paths without recomputing or interpreting prose.

The orchestration changes no acceptance threshold and performs no build itself. `run-matrix.mjs` remains the executor, so every generated screen or confirmation independently consumes the correctness gate, refuses CI, and applies the frozen child host policy.
