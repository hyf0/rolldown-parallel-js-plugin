# Scale-Crossover Protocol Amendment 6: MDX Rust-Pool and Aggregate CPU-Rate Policy

Status: frozen on 2026-07-12 before any Rust-pool allocation, CPU-rate calibration, or quota project timing is executed. This amendment makes the already specified MDX allocation and quota axes executable. It changes no corpus, prefix, lifecycle baseline, host gate, correctness gate, crossover statistic, resource threshold, or product-semantic requirement.

## Exact crossover dependency

Neither axis accepts copied scales, worker counts, summaries, or manually authored `passed` fields. Every generated matrix records the absolute path and SHA-256 of the base screen and every ordered confirmation/refinement artifact. Before generation, validation, execution, or summarization, the harness reloads those bytes, verifies current runner provenance, reruns the Amendment 5 decision, and requires exact mechanical and resource-acceptable crossover results. Allocation points are derived from the resource result's immediate repeated lower point, candidate, confirming next point, and full 9,157-source point, removing duplicates. Quota points are the resource crossover candidate and full corpus. The crossover reference also derives and freezes the project root, Rolldown root, correctness gate, host policy, and one normalized-output oracle for every selected scale from those exact artifacts; later matrices cannot supply or change their own execution template, and every policy child must match its scale's frozen oracle.

Every later policy matrix also records the path and SHA-256 of every preceding policy report. The generator reconstructs the only allowed next matrix from those reports and exact-compares it with the retained matrix. A missing, reordered, substituted, duplicated, or post-terminal artifact rejects the chain.

## Rust-pool allocation

The first allocation screen crosses Tokio worker counts `4, 8, 12, 18` with ordinary and JavaScript worker counts one through eight at every derived allocation point. Rayon remains 12 and the blocking pool remains 4. Ordinary is present at every Tokio setting. Each case is a fresh-process, uninstrumented, host-gated lifecycle-baseline run and is selection evidence only.

At each scale the screen selects a provisional resource-eligible Tokio/JavaScript pair using the frozen wall, CPU, RSS, and absolute-memory limits available from one sample; if none passes, it visibly falls back to the fastest screened worker pair. A screen cannot satisfy a bootstrap gate or establish an optimum. Both the screen-selected Tokio pool and the best different-pool runner-up therefore receive ten no-warmup rotated blocks of ordinary, their selected JavaScript counts, adjacent counts, worker four, and worker eight, with duplicates removed. Repeated selection uses candidate wall-median bootstrap intervals for the below-two-percent smaller-count tie rule and paired worker-to-ordinary speedup bootstrap only for gain and resource gates. Rayon screening consumes the repeated Tokio winner, not the one-shot winner.

Rayon counts `4, 8, 12` are then screened only for the JavaScript finalists actually repeated at the selected Tokio setting. Ordinary remains present at every Rayon setting, Tokio remains fixed per scale, and blocking remains 4. Both the selected Rayon pool and the best different-pool runner-up receive the same ten-block repeated confirmation. Configured Tokio, Rayon, blocking, and JavaScript capacities remain separate inputs; neither their sum nor any individual configured number is labelled observed CPU ownership or an operating-system quota.

## Aggregate CPU-rate controller

Quota generation requires a formally passed schema-2 calibration for the patched `opsengine/cpulimit` source at `f4d2682804931e7aea02a869137344bb5452a3cd`. The canonical upstream source tree contains 20 files and 55,759 bytes with SHA-256 `d7a8dccb84e90d854b146fb0b7363868e222c9f50469ebc11650f7165c76c21a`; the Apple patch SHA-256 is `de4c2800dbc1b4cbad8d280ec1aebecda7256dbf221fbb97daf83e7fa0a88060`; the frozen local binary SHA-256 is `233531824804f4be5ef3b425b0903bd36a90c069fd44598da4fad77e90eb0bd9`. Calibration also pins the exact calibration runner and load-target hashes.

Only the default formal calibration may pass: three rotated samples at 200%, 400%, 600%, and 800% with ten-second, eight-thread loads, plus five direct-versus-1,200% pairs on the pinned Node and M3 Pro host. Every target sample must remain within five percent, paired median wall and CPU ratios at 1,200% must remain within two percent, and every 1,200% controller record must have zero stop cycles and zero stopped duration. The median achieved CPU rate of those independent no-stop 1,200% pairs is the measured unconstrained saturation ceiling. Positive stop cycles are required only for targets below 95% of that measured ceiling; an admitted 800% sample may therefore have zero stops. Project generation rehashes the current source, patch, binary, and calibration harness, validates the frozen sample order and load shape, recomputes CPU rates, the saturation ceiling, level summaries, equivalence ratios and medians from raw observations, and validates every raw controller record; an artifact-authored summary cannot replace those checks.

The quota screen uses 400%, 800%, and 1,200% at the derived crossover and full points, with baseline Tokio 18, Rayon 12, blocking 4, and ordinary plus JavaScript worker counts one through eight. Each quota/scale confirmation includes ordinary, the screened best and adjacent counts, the unthrottled resource oracle count from the exact crossover chain, worker four, and worker eight, deduplicated, for ten no-warmup rotated blocks.

For quota execution, `/usr/bin/time -l` directly launches a Node policy child. Before importing the case runner, that Node process writes its PID and stops itself. The runner attaches the exact cpulimit binary to that PID; the controller resumes and duty-cycles the complete Node process, including its worker threads. The report requires the controller target PID to equal the child-reported Node PID, exactly one controller record, positive control cycles, stop cycles no greater than control cycles, stopped duration no greater than the directly measured Node wall, consistent stop-cycle/stopped-duration fields, stops at 400%, an achieved aggregate CPU rate no higher than 105% of the declared ceiling, direct `/usr/bin/time` wall/CPU/RSS nesting around the child interval, and the ordinary output/host/paging gates. The 800% quota may legitimately have zero stops at natural eight-worker saturation. Ready, controller-attachment, and whole-run deadlines use process-group cleanup that resumes a stopped Node, terminates it and the controller, escalates to kill after a bounded grace period, and reaps both. This arrangement avoids timing cpulimit in place of Node or leaving a stopped child behind.

Quota cases explicitly disable JavaScript and Rust instrumentation. Controller stop cycles and stopped duration are wall-policy evidence only; quota runs cannot enter service-latency, ready-width, worker initialization, JIT, or lifecycle attribution.

## Policy evidence and classification

Screens and confirmations remain separate evidence kinds and no policy raw report is conclusion-eligible by itself. Each summarizer emits `policyEvidenceByCase` with the standard per-variant wall median, total-process CPU median, peak-RSS median, resource eligibility, paired worker-to-ordinary wall-ratio bootstrap 95% upper bound when repeated, and selected resource oracle count. Every case also retains exact Rust-pool inputs and quota percentage so later evaluators can use stable JSON paths without treating configured capacities as observed CPU.
